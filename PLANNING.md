## 司机端核心逻辑

### 发布行程
司机输入：
- 出发地、目的地
- 出发时间
- 愿意绕路的最大时间（如 5 分钟 / 10 分钟）
- 可载客数（1–3 人）

发布后行程进入"待匹配"状态，对符合条件的乘客可见。

### 匹配筛选逻辑
系统推荐乘客的判断标准：
1. **绕路可接受** — 乘客上车点在司机原始路线的合理范围内（以实际驾车绕路时间/距离计算，非直线距离），不超过司机设定的阈值。
2. **方向一致** — 乘客目的地必须在司机行进方向上，反方向或大角度偏离不推荐（例：司机从 Georgetown → College Park，乘客目的地为 Bethesda 则不合适）。
3. **座位未满** — 已接受乘客数 < 司机设定上限。

### 司机操作流程
```
发布行程
  → 系统推送符合条件的乘客请求
  → 司机逐个查看（头像、学校、评分、上车点、目的地）
  → 接受 or 忽略
  → 接受后：双方信息互通，乘客收到通知
  → 出发时间到：行程开始
  → 完成后：双方互评，费用结算
```

### 费用计算逻辑
> 集中在单一文件管理，不散落各处。

**原则：司机不盈利只回本（合规关键，避免触发"营运车辆"法律问题）**

| 项目 | 说明 |
|------|------|
| 基准成本 | 司机原始路线里程 × IRS 标准里程费率 |
| 绕路附加 | 因接送乘客额外增加的里程单独计费 |
| 分摊规则 | 总费用由所有乘客均摊 |
| 平台抽成 | 从乘客支付中扣 10%，不从司机收 |

**示例：**
- 司机原路线 20 英里，接一名乘客多绕 2 英里，共 22 英里
- 费率 $0.21/英里 → 乘客应付 $4.62
- 加 10% 平台费 → 乘客实付 $5.08
- 司机实得 $4.62
